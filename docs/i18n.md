# Localization

`ani-gui` ships with four locales at MVP, chosen for markets where free-content tooling has the strongest organic demand:

| Locale | Code | Why this one |
|---|---|---|
| English | `en` | Lingua franca of anime fandom outside Japan |
| Brazilian Portuguese | `pt-BR` | Very large fandom; legal options expensive relative to local incomes |
| Latin American Spanish | `es-419` | Large fandoms across Mexico, Argentina, Chile, Peru; patchy legal catalog |
| Russian | `ru` | Crunchyroll exited Russia in 2022; legal options are sparse |

## How translation works

The frontend uses [Paraglide.js](https://inlang.com/m/gerre34r/library-inlang-paraglideJs). It compiles message bundles to type-safe TypeScript at build time вЂ” no runtime overhead, no missing keys at runtime, tree-shaken per-route.

## Adding a locale

1. Copy `gui/frontend/src/lib/i18n/en/` to a new directory keyed by the [BCP 47](https://www.rfc-editor.org/rfc/bcp/bcp47.txt) tag (e.g. `pl`, `tr`, `id`).
2. Translate every value. Keys remain identical to the English bundle.
3. Add the tag to `gui/frontend/project.inlang/settings.json` under `languageTags`.
4. Run `pnpm i18n:check`, which fails if any English bundle key is missing in the new locale.
5. Add a translation-attribution line to `docs/i18n.md` under the candidates table below.

The architecture is designed so adding a locale is purely a translation-file PR вЂ” no code changes required.

## What is and isn't translated

**Translated** вЂ” every UI string the user sees: labels, buttons, error messages, empty states, tooltips, settings, date/time formatting (via `Intl.*`).

**Not translated**:

- **Anime titles**: the user picks a title-language preference in Settings (English / Romaji / Native). The default differs by UI locale: `en` and `pt-BR` default to English (Portuguese metadata is sparse on Kitsu/AniList); `es-419` defaults to English; `ru` defaults to Romaji.
- **Provider names** (wixmp, hianime, filemoon, etc.) вЂ” proper nouns.
- **Genre tags** as raw strings вЂ” Kitsu returns them in English; we render them with locale-specific subtitling where applicable.

## RTL support

Out of scope at MVP. No Arabic or Hebrew locale is shipping yet. However, the codebase uses logical CSS properties (`margin-inline-start`, `padding-inline-end`, `text-align: start`) so adding an RTL locale becomes a translation-only PR + a `dir="rtl"` flip on `<html>`. The `tests/arch/i18n.sh` invariant test enforces no use of `margin-left` / `padding-left` / etc.

## Phase-2 candidate locales

Not blocking the MVP, but the i18n architecture supports adding any of these as translation-only PRs:

| Candidate | Code | Why eventually |
|---|---|---|
| Polish | `pl` | Large fandom, fewer legal options than Western Europe |
| Turkish | `tr` | Large youth fandom, weak legal Turkish-subbed catalog |
| Indonesian | `id` | Massive market, strong free-streaming culture |
| Vietnamese | `vi` | Large audience, weak legal availability |
| Ukrainian | `uk` | Similar dynamics to Russian, separate locale for cultural reasons |
| Arabic | `ar` | Significant audience; **requires RTL enablement first** |
